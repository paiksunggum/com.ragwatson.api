# Cursor 하네스 엔지니어링

> **트레이드오프:** 하네스는 모델을 느리게 보이게 할 수 있다. 대신 잘못된 가정·과잉 구현·부수적 수정·모호한 “완료”를 줄인다.

## 하네스가 막는 것

Karpathy가 반복적으로 짚는 패턴은 대략 네 가지다.

| 패턴 | 하네스에서의 대응 방향 |
| --- | --- |
| 조용한 가정 | 질문·대안 제시·트레이드오프 명시 |
| 요청보다 큰 코드 | 최소 구현, 불필요한 추상화·기능 금지 |
| 주변 코드까지 손대기 | 변경 범위를 요청에 고정 |
| “됐다”의 기준 부재 | 테스트·재현·빌드 등 검증 가능한 완료 정의 |

## 이 저장소에서의 하네스 구성

| 레이어 | 역할 |
| --- | --- |
| **규칙** (`.cursorrules`) | 에이전트에게 항상 주입되는 **짧은 운영 규칙**. 네 원칙을 명령형으로 압축한다. |
| **행동 지침** (`CLAUDE.md`) | 같은 원칙의 **전체 문장·예시·출처**. 규칙과 충돌 시 `CLAUDE.md`의 의도를 우선해 해석한다. |
| **맥락** | `@` 참조, 열린 파일, 컨텍스트에 넣은 문서. “무엇을 알려줬는지”가 곧 하네스의 입력이다. |
| **도구** | 검색·터미널·린트 등. **목표 중심 실행**에서 말하는 “검증”을 실제로 돌리는 쪽이다. |

하네스 엔지니어링은 위 요소를 **의도적으로 설계·갱신**하는 일이다. 규칙만 두고 검증 루프를 안 두면, 네 번째 원칙이 비어 있다.

## Karpathy 네 원칙 → Cursor 관행

### 1. 구현 전 사고 (Think Before Coding)

- 프롬프트가 애매하면 **코드를 쓰기 전에** 질문·가정 나열·해석 분기를 채팅에 남긴다.
- “일단 구현” 대신 **결정이 필요한지**를 먼저 판별한다.
- 장기적으로는 `CLAUDE.md`에 “모호할 때 멈춘다”는 문장이 규칙과 함께 작동하도록 유지한다.

### 2. 단순성 우선 (Simplicity First)

- 에이전트에게 **한 번에 하나의 목표**를 주는 편이 하네스가 단순해진다.
- 저장소 규칙에 “요청 밖 확장 금지”를 넣어, Cursor가 길게 퍼지는 출력을 제한한다.

### 3. 정밀한 수정 (Surgical Changes)

- `@파일`·범위를 명시해 **터치할 표면적**을 줄인다.
- 리뷰는 **diff가 요청과 직결되는지**만 먼저 본다. 스타일·리팩터는 별 요청으로 분리한다.

### 4. 목표 중심 실행 (Goal-Driven Execution)

- 작업 설명에 가능하면 **완료 정의**를 붙인다. (예: “`pytest` 전부 통과”, “해당 엔드포인트 curl로 200”)
- 다단계일 때는 채팅이나 이슈에 짧은 계획 + 단계마다 검증을 적는다.

```text
1. [단계] → 검증: [확인 사항]
2. [단계] → 검증: [확인 사항]
```

## 유지보수 체크

- **규칙과 본문이 어긋나면** `.cursorrules`를 짧게 고치고, 설명·예시는 `CLAUDE.md`에 맡긴다.
- 하네스가 과해지면(규칙만 비대해지면) **Karpathy 2번**에 따라 줄인다.

## 관련 파일

- `CLAUDE.md` — 네 원칙 전문
- `.cursorrules` — Cursor에 항상 줄 압축 규칙
